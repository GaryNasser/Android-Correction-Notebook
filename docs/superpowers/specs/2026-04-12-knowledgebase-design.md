# 知识库双页结构与 BITShare 下载集成设计

## 1. 目标

将当前静态占位的知识库页面重构为两个上方分栏页面：

- 文件管理页：以本地知识库目录为主，提供简洁、易用的层级管理能力
- BITShare 下载页：通过公开 API 搜索与下载资料，并直接保存到本地知识库

第一版目标是完成一个可用的 MVP：

- 用户可以在本地维护自己的知识库目录结构
- 用户可以搜索公开的 BITShare 资料
- 用户可以选择本地保存目录并将资料下载到知识库
- 下载完成后，文件以本地文件形式存在，同时保留来源元数据

## 2. 非目标

第一版不包含以下能力：

- BITShare 后台登录态与管理接口接入
- 远程目录镜像同步到本地
- 批量 ZIP 下载并自动拆分入库
- 文件内容预览器
- 多选批量移动、批量删除
- 跨设备同步
- 回传分享结果到 BITShare

## 3. 产品决策

### 3.1 页面结构

知识库页面改为两个 Tab：

- `文件管理`
- `BITShare 下载`

不保留当前的“错题库”“云端共享”结构。知识库模块的职责收敛为：

- 管理本地资料
- 获取远程资料

### 3.2 本地目录为主

知识库层级以本地自建目录为准，不跟随 BITShare 的远程目录结构。

BITShare 下载时，用户显式选择保存到哪个本地目录。这样可以保持知识库结构的可控性，避免远程目录体系污染本地组织方式。

### 3.3 入库策略

下载文件时同时落两类数据：

- 文件本体：保存到应用私有目录
- 元数据索引：保存来源、远程文件 ID、标题、路径、下载时间等信息

这允许后续扩展搜索、去重、导出、来源展示，而不需要重新设计数据层。

### 3.4 BITShare 接入范围

第一版只对接公开只读接口：

- `GET /api/public/search`
- `GET /api/public/files/{fileID}`
- `GET /api/public/files/{fileID}/download`
- 可选补充：`GET /api/public/files` 作为默认推荐列表

不接入 `/api/admin/...`，避免 Cookie 登录态、权限与跨域问题。

## 4. 页面设计

### 4.1 文件管理页

文件管理页采用移动端友好的逐级进入模型，而不是复杂树控件。

页面组成：

- 顶部路径栏：显示当前目录路径，例如 `知识库 / 数学 / 高数`
- 操作行：包含 `新建文件夹`、`排序`、`搜索当前目录`
- 内容列表：先展示子目录，再展示文件
- 空状态：目录为空时给出引导
- 最近下载入口：方便用户快速整理刚从 BITShare 导入的文件

交互规则：

- 点击目录进入下一级
- 点击文件打开操作菜单或详情底栏
- 支持文件操作：`重命名`、`移动`、`删除`、`导出/分享`
- 支持目录操作：`重命名`、`删除`

### 4.2 BITShare 下载页

BITShare 下载页采用搜索驱动结构。

页面组成：

- 顶部搜索框：输入关键词后执行远程搜索
- 排序切换：第一版保留 `综合`、`下载量`、`最新`
- 结果列表：展示标题、原始文件名、扩展名、大小、下载量、上传时间
- 详情面板：展示描述、来源路径、文件类型等信息
- 下载动作：`下载到知识库`

交互规则：

- 用户输入关键词后调用搜索接口
- 点击结果项查看详情
- 点击 `下载到知识库` 后弹出本地目录选择器
- 默认记住上一次成功保存的目录
- 下载成功后提示保存路径，并提供 `查看文件` 与 `继续搜索`

第一版不做批量下载，优先确保单文件下载链路稳定。

## 5. 数据模型

建议新增本地数据库，使用 `Room` 管理索引数据。

### 5.1 目录表 `kb_folder`

字段建议：

- `id`
- `name`
- `parentId`
- `createdAt`
- `updatedAt`

说明：

- 目录树仅表示本地知识库结构
- 根目录可通过 `parentId = null` 表示

### 5.2 文件表 `kb_file`

字段建议：

- `id`
- `folderId`
- `displayName`
- `storedName`
- `localPath`
- `mimeType`
- `sizeBytes`
- `sourceType`
- `sourceFileId`
- `sourceTitle`
- `sourcePath`
- `downloadedAt`
- `createdAt`
- `updatedAt`

说明：

- `displayName` 用于界面展示
- `storedName` 用于真实落盘，避免重名冲突
- `sourceType` 第一版固定为 `bitshare` 或 `manual`
- `sourceFileId` 对应 BITShare 文件 ID
- `sourcePath` 记录远程资料的路径或分类描述

## 6. 本地存储结构

真实文件保存在应用私有目录，例如：

`files/knowledge_base/...`

目录与数据库需要保持一致，但不要求真实文件名与显示名完全相同。推荐使用以下策略：

- 显示层使用 `displayName`
- 文件系统使用稳定唯一的 `storedName`

这样在重命名文件时，可以只更新显示名称；如果后续需要同步物理文件名，也有清晰的迁移路径。

## 7. 模块划分

建议在现有项目结构下新增以下边界：

### 7.1 BITShare 远程层

- `BitShareApiService`
- `BitShareRemoteManager` 或 `BitShareRepository`
- 只封装公开接口请求与响应映射

职责：

- 搜索资料
- 获取资料详情
- 下载文件流

### 7.2 本地知识库数据层

- `KnowledgeBaseFolderEntity`
- `KnowledgeBaseFileEntity`
- `KnowledgeBaseDao`
- `KnowledgeBaseDatabase`
- `KnowledgeBaseRepository`

职责：

- 目录读写
- 文件元数据读写
- 目录级搜索与排序
- 文件移动、重命名、删除的一致性控制

### 7.3 文件存储层

- `KnowledgeBaseFileStorage`

职责：

- 创建目录
- 写入下载文件
- 删除本地文件
- 移动物理文件
- 导出分享时提供 `Uri`

### 7.4 表现层

- `KnowledgeBaseViewModel`
- `KnowledgeBaseUiState`
- `KnowledgeBaseEvent`

建议将“文件管理”和“BITShare 下载”放在同一个 ViewModel 下的两个子状态，或拆成两个子 ViewModel，但共享同一个 repository 层。

第一版更推荐单页面 ViewModel + 子状态结构，减少页面间同步复杂度。

## 8. 核心数据流

### 8.1 搜索流程

1. 用户在 BITShare 页输入关键词
2. ViewModel 发起远程搜索
3. Repository 调用 `GET /api/public/search`
4. 结果映射为 UI 模型并展示

### 8.2 下载入库流程

1. 用户在 BITShare 搜索结果中点击下载
2. 系统弹出本地目录选择器
3. 用户确认保存目录
4. Repository 调用 `GET /api/public/files/{fileID}` 获取详情
5. Repository 调用 `GET /api/public/files/{fileID}/download` 获取文件流
6. 文件存储层将文件写入应用私有目录
7. 本地数据库写入 `kb_file` 元数据
8. 文件管理页刷新并显示新文件

### 8.3 本地管理流程

1. 文件管理页读取当前目录下的子目录与文件
2. 用户进行新建目录、重命名、移动、删除等操作
3. Repository 先更新数据库，再按需同步文件系统
4. 若文件系统操作失败，则停止提交最终状态并向 UI 返回错误

## 9. 异常处理

第一版需要明确覆盖以下错误场景：

- 搜索失败：保留当前关键词并提示重试
- 下载失败：不生成数据库记录，不留下不完整文件
- 写入失败：提示存储失败，并允许重新下载
- 远程资源失效：对 `404`、`410` 提示资源不可用
- 重名冲突：自动在显示名后追加 `(1)`、`(2)` 等后缀
- 目录缺失：若目标目录已被删除，提示重新选择保存位置

原则：

- 不允许出现数据库有记录但本地没有文件的脏状态
- 不允许出现写入了一半的临时文件长期残留

## 10. UI 状态

两个页面都需要完整状态建模。

### 10.1 文件管理页状态

- 初始加载中
- 空目录
- 正常列表
- 搜索中
- 操作中
- 操作失败

### 10.2 BITShare 下载页状态

- 空搜索态
- 搜索加载中
- 搜索成功
- 搜索无结果
- 搜索失败
- 下载中
- 下载成功
- 下载失败

## 11. 排序与搜索

第一版建议能力收敛如下：

### 11.1 文件管理页

- 当前目录内搜索
- 排序方式：名称、最近更新

### 11.2 BITShare 页

- 按关键词搜索
- 排序方式：
  - 综合
  - 下载量优先
  - 最新优先

如果公开接口无法直接提供“最新”或“综合”映射，则在 ViewModel 中统一回退到接口支持的默认行为，并保持 UI 文案与实际行为一致。

## 12. 导出与分享

文件管理页需要支持将本地文件导出或分享给其他应用。

实现上建议：

- 通过 `FileProvider` 暴露只读 `Uri`
- UI 提供 `导出/分享` 入口
- 第一版只支持单文件导出

目录导出不纳入第一版范围。

## 13. 测试策略

### 13.1 单元测试

- 搜索结果响应映射
- 下载成功后的元数据入库
- 下载失败时不写脏数据
- 重名冲突生成逻辑
- 目录移动与文件移动路径计算

### 13.2 集成测试

- 新建目录后可立即进入
- 下载文件后在目标目录可见
- 删除文件后数据库与本地文件均移除
- 重命名后列表正确刷新

### 13.3 UI 测试

- Tab 切换
- 空状态展示
- 搜索失败态提示
- 下载成功提示与跳转入口

## 14. 实施顺序建议

建议按以下顺序实现：

1. 重构知识库页面为双 Tab
2. 引入本地目录与文件索引模型
3. 打通文件管理基础操作
4. 接入 BITShare 搜索接口
5. 接入下载并完成本地入库
6. 补齐导出分享与错误状态
7. 增加测试

## 15. 风险与约束

- BITShare 更适合服务端到服务端调用；安卓客户端直连公开接口可行，但要关注网络可达性与证书兼容性
- 第一版不依赖后台 Cookie 登录态，避免会话管理复杂度
- 文件系统与数据库双写是核心一致性风险点，需要通过 repository 串联控制
- 若项目当前未接入 `Room`，需要新增数据库依赖与迁移初始化

## 16. 验收标准

以下能力全部成立时，视为第一版完成：

- 用户可以在文件管理页创建和浏览本地目录
- 用户可以对本地文件执行重命名、移动、删除、导出
- 用户可以在 BITShare 页搜索公开资料
- 用户可以选择本地目录并下载单个资料到知识库
- 下载完成后文件立即出现在目标目录中
- 下载文件带有可查询的来源元数据
- 搜索失败、下载失败、重名冲突等关键异常有明确反馈且不会产生脏数据
